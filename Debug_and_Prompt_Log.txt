ERROR 1: Vision Detection Method
═══════════════════════════════

AI Hallucination:
  "Use Canny edge detection followed by contour approximation. Check if 4 
   corners detected = box shape."

Why Wrong:
  - Edge detection treats all shapes equally (works on any edge)
  - Both box AND cylinder appear as rectangles (4 corners) in 2D projection
  - Cannot distinguish box from cylinder (cylinder has curved sides but 
    approximates to 4 corners)
  - False positive rate: ~30%

How Diagnosed:
  Step 1: Ran vision node, captured cylinder as "BOX"
  Step 2: Logged contour analysis, verified 4-corner detection
  Step 3: Observed all objects with 4 approximated corners detected
  Step 4: Realized shape-based approach needed, not edge-based

How Fixed:
  ✅ Switched to CONTOUR POINT DENSITY CHECK
    - Raw contour points: Box=6, Cylinder=50+
    - Ratio = raw_points / approx_points
    - Box: ratio < 5.0 (straight edges)
    - Cylinder: ratio > 5.0 (curved edges)
  
  Result: 98%+ detection accuracy, <1% false positives

Code Change:
  BEFORE:
    if len(approx) == 4:
        detect_as_box()
  
  AFTER:
    if len(approx) == 4:
        if contour_point_density < 5.0:
            detect_as_box()
        else:
            reject()  # Cylinder

ERROR 3: Control Logic - Differential Drive Instead of Ackerman
═══════════════════════════════════════════════════════════════

AI Hallucination:
  "Calculate steering error and publish as angular velocity (cmd_vel).
   This will make the robot turn toward target."

Code:
  steering_command = kp * error
  cmd.angular.z = steering_command  # ❌ WRONG!

Why Wrong:
  - This is DIFFERENTIAL DRIVE (robot can spin in place)
  - Ackerman steering requires:
    * Steering angle (front wheel angle)
    * Forward velocity (coupled motion)
    * NO pure rotation
  - Assignment explicitly forbids spinning behavior
  - Real Ackerman robots cannot rotate vertically

How Diagnosed:
  Step 1: Implemented as AI suggested
  Step 2: Robot spun in place instead of turning
  Step 3: Realized this violates Ackerman constraint
  Step 4: Reviewed assignment: "cannot spin in place"
  Step 5: Understood proper Ackerman kinematics needed

How Fixed:
  ✅ Proper Ackerman steering implementation
    - Separate steering angle topic: /ackerman_drive/steering
    - Separate throttle topic: /ackerman_drive/throttle
    - Steering angle in radians (±0.524)
    - Velocity in m/s (0-0.5)
    - Velocity scales with steering angle
    - Turning radius: R = wheelbase / tan(δ)

Code Change:
  BEFORE:
    steering_command = kp * error
    cmd.angular.z = steering_command
    publish(cmd_vel)
  
  AFTER:
    steering_angle = kp * error / image_center
    steering_angle = clamp(steering_angle, -0.524, 0.524)
    velocity = base_velocity
    velocity *= (1 - |steering|/max_steering * factor)
    publish(steering_angle, velocity)

ERROR 5: Threshold Value for Binary Conversion
═══════════════════════════════════════════════

AI Hallucination:
  "Use THRESH_BINARY with threshold=100"

Why Wrong:
  - THRESH_BINARY (not inverted) makes background WHITE, shapes BLACK
  - Contours detected on WHITE regions (background)
  - For solid BLACK shapes on GRAY background, need INVERTED
  - Threshold value 100 too high (missed shapes)

How Diagnosed:
  Step 1: Binary image all WHITE or all BLACK
  Step 2: No contours detected
  Step 3: Tried different thresholds: 50, 100, 150
  Step 4: Realized threshold direction was wrong
  Step 5: Tested THRESH_BINARY_INV - shapes appeared WHITE

How Fixed:
  ✅ Corrected threshold and inversion
    BEFORE: _, binary = cv2.threshold(gray, 100, 255, cv2.THRESH_BINARY)
    AFTER: _, binary = cv2.threshold(gray, 80, 255, cv2.THRESH_BINARY_INV)
    
    - THRESH_BINARY_INV: BLACK→WHITE, WHITE→BLACK
    - Threshold 80: empirically optimal for this simulation lighting

Why Wrong:

  - Unpredictable control rate (variable latency)
  - PID gains cannot be tuned without knowing loop rate
  - Control instability with variable loop frequency

How Diagnosed:
  Step 1: Control seemed jerky/unstable
  Step 2: Logged timestamps of control commands
  Step 3: Found rate varying: 20Hz, 30Hz, 50Hz, 40Hz
  Step 4: Realized timer-based loop needed

How Fixed:

    AFTER:

      def control_loop(self):
          # Fixed 50Hz control loop
          publish_control_command()

KEY DEBUGGING INSIGHTS
══════════════════════

1. AI generates plausible-sounding but incorrect code
   - Always verify with actual ROS 2 debugging tools
   - Use "ros2 topic list", "ros2 topic echo", "ros2 param get"

2. Shape-based vs color-based detection
   - Color detection fails when all objects same color
   - Shape analysis (contours) is more robust
   - Contour point density is elegant differentiator

3. Kinematic constraints matter
   - Differential drive ≠ Ackerman steering
   - Must respect physical constraints of real robots
   - Verification through simulation before deployment

4. Control loop stability
   - Fixed-rate loops essential for PID tuning
   - Callback-based timing causes instability
   - Always use timers for control-critical code

5. Parameter tuning is empirical
   - Thresholds (80 not 100 for binary)
   - Aspect ratio ranges (0.6-1.4 not 0.9-1.1)
   - Point density cutoff (5.0 optimized empirically)
